# π0.5: Hierarchical Vision-Language-Action Flow Model

**Organization:** Physical Intelligence
**Year:** 2025

---

# Problem

Early Vision-Language-Action (VLA) models such as RT-1, RT-2 and π0 demonstrate that robot policies can be learned from large datasets using transformer-based architectures.

However, these systems still struggle with **long-horizon manipulation tasks**.

Typical problems include:

- tasks requiring many sequential steps
- complex household environments
- generalization to new homes or scenes
- difficulty maintaining coherent plans over long time horizons

Low-level policies alone often fail because they attempt to directly map observations to actions without an explicit notion of **task structure**.

This raises a key question:

> Can robot policies benefit from explicit **high-level planning** combined with continuous action generation?

---

# Why it matters

Many real-world robotic tasks involve hierarchical structure.

For example:

make coffee

may involve:

1. find mug
2. place mug under machine
3. press button
4. wait for coffee

Without modeling this structure, a policy must implicitly discover long chains of actions.

This makes training difficult and often reduces generalization.

Introducing **high-level subtasks** could help:

- stabilize long-horizon planning
- improve generalization to new scenes
- allow models to reason about task structure

---

# Attempt

π0.5 extends π0 with a **hierarchical architecture**.

The key idea is to separate behavior into two levels:

1. **High-level subtask prediction** (language-like output)
2. **Low-level action generation** (continuous flow-based control)

The system also introduces:

- FAST action tokenization for efficient pretraining
- a two-stage training pipeline
- multimodal token routing for different input types

In short, π0.5 combines:

- language generation
- action-token pretraining
- continuous flow matching
- hierarchical planning

---

# Model

## Multimodal Token Input

The model processes a sequence of multimodal tokens.

Each token may represent:

- text instruction tokens
- image patches
- robot state observations
- noisy action inputs used for flow matching

Formally a token (x_i) may be:

- text token (x_i^w)
- image patch (x_i^I)
- action value (x_i^a)

The observation and instruction form the **prefix of the token sequence**.

---

## Token-Type Routing

A routing function (\rho(x_i)) determines which encoder processes a token.

Examples:

- text tokens → embedding matrix
- image tokens → vision encoder
- action tokens → robotics action expert

This allows the system to reuse vision-language modeling while introducing robotics-specific processing.

---

## Two-Stage Output Structure

π0.5 predicts two types of outputs.

1. **High-level plan tokens**
2. **continuous robot actions**

The output sequence is split as:

- first (M) outputs → language token logits
- last (H) outputs → action outputs

The predicted language tokens represent a **subtask plan**.

These plans condition the generation of continuous robot actions.

---

## Flow Matching for Continuous Actions

Like π0, π0.5 generates actions using **conditional flow matching**.

During inference the model performs:

- ~10 denoising steps
- conditioned on observations and predicted subtasks

This produces smooth continuous action trajectories.

---

# FAST Action Tokenization

π0.5 introduces **FAST** to accelerate pretraining.

The goal is to convert continuous action trajectories into discrete tokens.

Steps:

1. normalize each action dimension to [-1,1]
2. apply **Discrete Cosine Transform (DCT)**
3. flatten the representation
4. train a **Byte Pair Encoder (BPE)** to compress it

This produces discrete action tokens that can be used during pretraining.

FAST allows the model to learn action structure earlier during training.

---

# Training

π0.5 uses a **two-stage training pipeline**.

## Stage 1: Pretraining

During pretraining the model behaves like a standard vision-language model.

The model predicts tokens for:

- text
- object locations
- FAST action tokens

Training length:

- ~280k gradient steps

Additional supervision may include:

- subtask annotations
- object localization
- captioning
- question answering

---

## Stage 2: Post-training

During post-training the robotics action expert is activated.

Key changes:

- flow matching loss enabled
- additional expert weights
- filtered robot trajectories

Training continues for:

- ~80k additional steps

---

# Loss

The training objective combines two components.

[
\mathcal{L} = \mathcal{H} + \alpha \mathcal{F}
]

Where:

- (\mathcal{H}) = cross-entropy loss for token prediction
- (\mathcal{F}) = flow-matching loss for actions

This jointly trains:

- language planning
- continuous robot control

---

# Failure / Limitation

Despite improvements, several challenges remain.

### Data requirements

Hierarchical models require rich annotations such as subtasks and language labels.

### Training complexity

The system combines multiple components:

- token prediction
- flow matching
- hierarchical outputs

which increases training complexity.

### Dataset mixture sensitivity

Performance depends heavily on the composition of the training data.

---

# Key Idea

π0.5 introduces **hierarchical VLA policies**.

Instead of directly mapping observations to actions, the system first predicts a high-level subtask and then generates continuous actions conditioned on that plan.

---

# Takeaway

π0.5 moves VLA systems toward a more structured policy design:

perception → high-level plan → continuous control

By combining hierarchical planning with flow-based action generation, π0.5 aims to improve long-horizon reasoning and generalization in real-world robotic tasks.
