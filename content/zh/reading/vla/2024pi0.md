# π0：用于通用机器人控制的 Vision-Language-Action Flow 模型

**机构**：Physical Intelligence
**年份**：2024
**论文**：[https://www.physicalintelligence.company/download/pi0.pdf](https://www.physicalintelligence.company/download/pi0.pdf)

---

# Problem

当前许多 Vision-Language-Action (VLA) 模型将机器人动作表示为 **离散 token**。

例如：

- RT-2
- OpenVLA
- Octo

这种做法虽然能够复用语言模型架构，但也带来了一些问题：

- 动作离散化可能降低控制精度
- token 生成可能产生不平滑或不自然的轨迹
- 连续控制问题被强行转化为符号预测

而在真实机器人控制中，动作本质上是 **连续轨迹（continuous trajectory）**。

因此一个关键问题是：

> 是否可以在 VLA 框架中直接生成 **连续动作轨迹**，而不是离散动作 token？

---

# Why it matters

通用机器人策略通常需要同时具备三种能力：

1. **语义理解能力**
   理解语言指令、物体类别以及任务语义。

2. **连续控制能力**
   生成平滑且稳定的动作轨迹。

3. **跨机器人泛化能力**
   在不同机器人本体（embodiment）之间迁移。

大型 Vision-Language Model 已经具备强大的语义理解能力，但它们并不擅长生成连续控制信号。

另一方面，传统机器人策略可以生成连续动作，但缺乏语义知识。

因此自然的问题是：

> 是否可以结合 **互联网规模语义知识 + 连续机器人控制**？

---

# Attempt

π0 提出一种新的思路：

**使用 flow matching 来生成连续机器人动作。**

整体系统由三个核心组件组成：

- 提供语义理解能力的 **Vision-Language Model (VLM)**
- 负责机器人控制的 **action expert**
- 用于连续动作生成的 **flow matching**

与 token-based VLA 不同，π0 并不预测离散动作符号，而是通过 **连续去噪过程** 生成动作轨迹。

---

# Model

## VLM Backbone

π0 的语义 backbone 基于 **PaliGemma**，而 PaliGemma 又建立在 **Gemma** 之上。

论文中给出的基础架构包括：

- multi-query attention
- width: 2048
- depth: 18
- MLP dimension: 16384
- heads: 18
- KV heads: 1
- head dimension: 256

π0 使用的是更小的配置：

- width: 1024
- MLP dimension: 4096
- 约 **3B 参数**

可以近似理解为：

SigLIP (vision encoder)

- Gemma (language model)

图像通过 ViT-style encoder 得到 embedding，然后与语言 token 一起进入 VLM。

---

## Inputs

模型输入包括：

- 来自 **2–3 个相机** 的图像观测 (I_i^t)
- 语言指令 (l_t)
- 本体感觉状态 (q_t)

例如某个 embodiment 中：

(q_t) 共有 **18 维**，包括：

- 两个 6-DoF 机械臂
- 两个 gripper
- 一个 AGV
- 一个垂直移动主体

这些状态会被投影到与语言 token 相同的 embedding 空间。

---

## Action Expert

π0 引入了一个机器人专用模块：

**action expert**。

模型参数分为两部分：

- **VLM 权重**：处理视觉与语言
- **action expert 权重**：处理机器人状态与动作

两者共享一个 Transformer，但负责不同计算路径。

这样可以：

- 复用互联网语义知识
- 保留机器人控制专用模块

---

## Action Horizon

模型预测的是 **动作轨迹** 而不是单步动作：

[
A*t = [a_t, ..., a*{t+H-1}]
]

其中：

[
H = 50
]

训练时模型接收 **带噪声的动作轨迹** 并学习去噪。

---

## Flow Matching

π0 使用 **conditional flow matching** 生成连续动作。

模型学习一个 velocity field，将噪声逐步转换为有效动作轨迹。

论文通过一个 MLP 将 timestep (\tau) 注入模型：

[
W_3 \cdot swish(W_2 \cdot concat(W_1 a'_{t,\tau}, \phi(\tau)))
]

这样 action expert 能同时感知：

- 当前动作噪声
- flow timestep

---

## Attention Mask

π0 使用 **blockwise causal attention mask**。

Token 顺序：

1. 图像 / 语言 token
2. 机器人状态 token
3. 动作 token

Attention 规则：

- 当前 block 内允许双向 attention
- 可以访问前一个 block
- 不能看到未来 block

这种设计既保持因果结构，又增强局部条件信息。

---

# Failure / Limitation

### 数据需求仍然较高

跨 embodiment 数据非常昂贵且难以收集。

目前仍不清楚每个任务需要多少数据才能稳定训练。

---

### 跨机器人泛化仍然困难

不同机器人具有不同：

- 运动学结构
- 控制接口
- 观测空间

真正通用的机器人策略仍然是开放问题。

---

### Flow 推理计算成本较高

Flow matching 在推理阶段需要 **多步去噪**，计算成本可能高于 token 预测。

---

# Results

π0 的一些关键观察包括：

- 可以学习 **长时序任务**（最长约 **10 分钟**）
- 连续 flow 生成的动作更加平滑
- 评估指标使用 **0–1 partial success score**

训练数据来源包括：

- Open-X Embodiment
- DROID
- Bridge
- 内部跨 embodiment 数据

总体覆盖：

- **7 种 robot embodiment**
- **68 个任务**

---

# Takeaway

π0 提出一个重要方向：

> 机器人动作更适合建模为 **连续轨迹生成问题**，而不是离散 token 预测。

在 π0 中：

- VLM 提供语义理解
- action expert 负责机器人控制
- flow matching 生成平滑连续轨迹

这种设计让 VLA 模型更加接近 **真实机器人控制系统**。
